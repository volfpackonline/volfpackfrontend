1 Vision & problem

The problem

Text-to-video is exploding, but the tooling is closed, expensive, and lock-in heavy. Runway, Pika, Luma, Sora, and Kling are all proprietary SaaS: your prompts, assets, and spend live on someone else's platform, metered at their margin. Developers and technical creators who want to use their own provider keys, self-host for privacy/cost, swap models as the frontier moves, and build on top of a hackable codebase have no good open-source home. VolfPack is that home.

The vision

The open-source control plane for text-to-video. Describe a scene, pick a model, generate — on infrastructure and keys you control. One studio UI, every video model behind one contract.

Why now

  • Frontier video models expose stable HTTP APIs (Runway Gen-3/4, Luma, Pika, Veo, Kling, Sora).
  • Open-weight video models are viable to self-host (Mochi-1, LTX-Video, HunyuanVideo, Wan 2.x) on a single high-VRAM GPU or via Replicate/Fal.
  • "BYO-key" + "open-core" is a proven OSS business pattern.

3 Goals, scope & non-goals

v1 Goals

  1. A self-hostable studio: marketing site + Studio UI + Python backend.
  2. Provider-agnostic generation across ≥3 video + ≥3 LLM providers.
  3. BYO-key default; optional managed keys designed-in.
  4. Async job lifecycle with progress, history, asset storage.
  5. One-command local run (docker compose up) + documented deploy.

Out of scope for v1

Video editing/timeline, audio/voiceover, mobile native, fine-tuning, real-time collab, billing/payments (designed-for, not built).

Non-goals (philosophy)

  • We are not a model lab — we don't train video models.
  • We are not a key-reseller — BYO-key is first-class; managed is opt-in.
  • We won't add a loud second brand color (per design.md). 🙂

4 ⚑ Central decision — keys & providers

This is the most important section. Everything downstream depends on it.

The headline "works on any LLM key" is technically incomplete. LLM provider keys (OpenAI, Anthropic, Gemini, …) do not generate video. Video pixels come from video-model providers. VolfPack therefore has two distinct provider classes behind one abstraction:

ClassPurposeExample providers
LLM providersPrompt enhancement, scene scripting, storyboard text, moderation, captions/SEOOpenAI, Anthropic, Gemini, Mistral, Groq, Ollama (local)
Video providersThe actual text/image-to-video renderRunway, Luma, Pika, Veo, Kling, Replicate/Fal (Mochi, LTX-Video, Hunyuan, Wan)

A single generation may use both: LLM expands the prompt → video provider renders it.

Provider abstraction

  • VideoProvider interface: submit, poll, cancel, capabilities().
  • LLMProvider via LiteLLM (one API, ~100 providers, BYO-key).
  • Registry/router: maps model id (e.g. runway:gen3a_turbo) → adapter, validates capabilities, routes, supports fallback chains.

Key handling modes (both designed-in)

  1. BYO-key (default, self-host): keys in server .env or, multi-user, encrypted per user/org (envelope encryption). Calls server-side; keys never touch the browser.
  2. Managed (optional, hosted): VolfPack keys, metered for billing. Flag-gated, off by default.
Principle: the request/response contract is identical regardless of key mode. Key mode is a deployment concern, not a product surface.

4.4 Local / on-device generation (no keys, no cloud)

A third provider category sits alongside cloud video: local open-weight models on the user's own GPU — no API key, no per-call cost, fully private. A first-class path for self-hosters who never want to call an external API. Local is just another VideoProvider; the contract is unchanged.

  • ComfyUIProvider (recommended): user runs ComfyUI (the de-facto OSS local video engine); VolfPack talks to its HTTP API + websocket for progress. ComfyUI owns GPU/VRAM/downloads. Supports LTX-Video, Mochi-1, HunyuanVideo, Wan 2.x, AnimateDiff.
  • DiffusersProvider (self-contained): a Python worker loads models via 🤗 diffusers directly — CUDA on NVIDIA, MPS on Apple Silicon, CPU fallback. More integrated, heavier to maintain — ship after ComfyUI.

VRAM-aware gating: each local model declares its memory need; VolfPack detects host VRAM/unified memory and enables / warns / disables each model — no silent OOM on weak hardware, everything unlocked on strong hardware.

Available VRAM / unified memoryRealistically runnable locally
~8 GB (M1 Air, GTX 1070)AnimateDiff only (short, low-res). Recommend cloud/API instead.
12–16 GB (RTX 3060/4060Ti, M2/M3 16GB)LTX-Video (fast), AnimateDiff — the local sweet spot
24 GB (RTX 3090/4090, M-series 32GB+)+ Mochi-1, quantized HunyuanVideo / Wan
40 GB+ (A100/H100, multi-GPU)Everything, full quality
Honest constraint: local is VRAM-bound and slower than cloud APIs (open models are CUDA-tuned; Apple MPS lags). Local is a privacy/cost win, not a speed/quality win — the UI sets that expectation. Stance: VolfPack ships the capability; whether a model runs depends on the user's hardware — exactly the Open WebUI / ComfyUI pattern.

4.5 What we provide — not an LLM wrapper

A fair challenge for any AI product: "Isn't this just a thin UI over someone else's API?" This section answers it head-on — if VolfPack were deleted, what would users lose that they couldn't rebuild with a fetch()?

What VolfPack actually provides

#We provideWhy it's real work, not a wrapper
1Provider-agnostic abstraction over cloud APIs and local modelsEvery provider has a different request shape, polling model, capabilities, error format. We unify ~8+ that disagree; a wrapper hits one.
2Capability registry + validationWe reject impossible requests (duration/ratio/res/VRAM) before spending money/time. A wrapper just forwards and lets the provider 400.
3Durable async job orchestrationQueue, workers, at-least-once + idempotent submit, progress, restart-survival. A real distributed-systems problem, not an API call.
4Local/on-device generation (§4.4)ComfyUI/Diffusers + VRAM gating. There is no API to wrap — it runs on the user's own GPU.
5Self-hostable, private deploymentYour infra, keys, data. A wrapper's whole value (convenience) is the opposite of this.
6Secure multi-tenant key custodyEnvelope-encrypted per-user keys, never logged/returned/browser-exposed. A wrapper has no key-mgmt problem.
7Fallback chains + routingProvider down → auto-retry on fallback; route by cost/speed/quality across providers.
8Cost attribution & meteringPer-job cost across heterogeneous units (seconds vs tokens vs credits), budget caps, transparency.
9Open, hackable adapter SDK~100-line pattern to add any model. Community extends VolfPack; you can't extend a closed wrapper.
10Persistent studio: history, re-open, variations, storyboard, asset lifecycleState + workflow the providers don't give — they're stateless render endpoints.
11Real product UX + design systemStudio, marketing, accessibility, brand. A product surface, not a Postman collection.

The anti-wrapper test

A thin wrapper has all of these; VolfPack has none:

Thin wrapperVolfPack
✗ Wraps exactly one provider✓ Abstracts many (cloud + local)
✗ Stateless pass-through✓ Owns jobs, history, assets, retries
✗ No logic between user and model✓ Validates, enhances, moderates, routes, attributes cost
✗ Breaks when the provider changes✓ Isolated adapters + contract tests absorb churn
✗ Must use their infra/keys✓ Self-host, BYO-key, or fully local
✗ Closed; you adapt to it✓ MIT + adapter SDK; it adapts to you

The defensible moat (be honest — see §4.6)

Blunt truth: most of the table above is features, not moats. Abstraction, queues, the UI, capability registries — all good engineering, but a competent team copies them in weeks, and our MIT license makes the code copyable by definition. Code is not a moat.

The one durable, technical moat is the orchestration intelligence layer (§4.6): smart routing, per-model prompt translation, cross-shot consistency, and quality scoring that compound with usage data we accumulate and a late-arriving competitor cannot. Everything else is a feature; this is the moat.

One-line framing: A wrapper makes one model easier to call. VolfPack makes every model work the same way — and over time, makes them work better than calling them directly, by learning which model + which prompt gives the best result.

4.6 The actual tech moat — orchestration intelligence

The honest answer to "what's the defensible technology?" It is not the abstraction layer (copyable) — it's the intelligence on top of it that compounds with data.

Why this is defensible when abstraction isn't

Provider abstraction (not a moat)Orchestration intelligence (the moat)
NatureOne-time engineering costCompounding data + R&D asset
Copyable?Yes — reimplement in weeksNo — code ≠ accumulated routing data
Improves with usage?No, staticYes — every generation gets smarter
Late competitorCatches up fastPermanently behind on data

The four intelligence layers (build order)

  1. Per-model prompt translation. Each model wants different phrasing; we learn (LLM-assisted + empirical) to rewrite one intent into the optimal prompt per model. Earned in data, not code.
  2. Cost / quality / speed routing. Given prompt + budget + deadline, pick the best model — and learn from outcomes which wins for which request.
  3. Quality scoring + auto-retry. Score outputs (heuristics → learned → optional LLM-as-judge), retry low scores elsewhere. Builds the labeled dataset powering layers 1–2.
  4. Cross-shot consistency. Stable character/style across clips (seed lock, reference conditioning, latent reuse, LLM-anchored continuity) — a genuinely unsolved hard problem.

The compounding-data flywheel

more users → more generations → more (prompt, model, result, score) data
   → smarter routing + prompt translation + consistency
   → better results than calling the provider directly
   → more users  ↺

The moat is the flywheel, not any single algorithm. A competitor forking our MIT code on day 1 still starts with zero outcome data; we start with all of ours. The gap widens with every generation.

Honest caveat (don't overclaim)

This moat does not exist at launch. v1 ships the architecture (routing hooks, score capture, prompt-translation interface); the intelligence is thin until usage accumulates. That's fine — and is why it's defensible: a moat you can build on day one is one anyone can copy on day one. We design for it now (data capture from the first generation) and it sharpens over time.

⚠ The fatal contradiction in this moat

The data flywheel conflicts directly with our own privacy/self-host pitch. Self-hosted users' data stays on their server — we never see it (the entire point of §4.4 & §11). So the flywheel only spins on the hosted tier (post-v1, optional, competing with the free self-host version we tell everyone to use). We've designed a moat structurally incompatible with our distribution strategy. Unresolved options:

  1. Opt-in anonymized telemetry — beg self-hosters to share data (most won't).
  2. Moat lives only on the hosted tier — self-host is a loss-leader funnel.
  3. Drop the data-flywheel claim — admit there's no compounding tech moat; compete on execution + ecosystem (§14.5).

Until resolved, treat §4.6 as aspirational, not proven. See §14.5.

14.5 Why this project might fail (devil's advocate)

A PRD that can't argue against itself isn't trustworthy. This is the adversarial read — the case a skeptical investor or hostile HN thread makes. Each point is real; some may be fatal. Read this before believing §4.6.

W1 — The moat contradicts the strategy most serious

The data-flywheel moat (§4.6) needs usage data; the privacy/self-host pitch (§4.4, §11) guarantees we never see self-hosted usage data. Moat and distribution are mutually exclusive. As designed, there is likely no compounding tech moat for the product people actually adopt (self-host).

W2 — "Orchestration intelligence" may solve a non-problem

  • Prompt translation bets on models staying diverse — but they're converging on good NL understanding. The skill may obsolete itself.
  • Multi-provider routing assumes users want to optimize across many models. Most pick one they trust. Auto-routing may be a solution without a problem.
  • Cross-shot consistency is being solved inside the models by labs with billions. Out-innovating Runway/Google solo isn't credible.

W3 — The economics punish the exact target user

Video gen is expensive and slow. The price-sensitive dev who wants BYO-key is the least likely to keep paying for generations. Most drawn to self-host = least likely to generate at volume.

W4 — Local gen (our best anti-wrapper point) needs a $1,500+ GPU

The strongest "not a wrapper" claim is usable only on high-VRAM hardware a tiny minority owns. The dev machine here (8 GB M1) literally can't run the good models. The differentiator reaches a sliver of the market.

W5 — The incumbents are also the studios

Runway, Pika, Luma all ship polished web apps. Our aggregation value evaporates the moment a user decides one model is "good enough" — the common case. We're a thin layer between users and labs building the same UX themselves.

W6 — Provider dependency & ToS risk (underweighted in R5)

Closed providers can forbid third-party UIs or kill BYO-key access overnight. The "aggregate the closed APIs" pitch sits entirely at their mercy.

W7 — The community moat assumes a community

The adapter-ecosystem story needs contributors. Most OSS projects never get a second one. Meanwhile a solo maintainer chases ~8 churning video APIs — a treadmill.

W8 — Honest verdict

This is a strong learning/portfolio project and a plausible small, beloved OSS tool ("the Open WebUI for video"). It is a weak startup and a very weak defensible business. Best realistic outcome: a respected MIT community tool that's hard to monetize — not a venture-scale company with a tech moat.

What would make it not bad (pivots worth considering)

  • Niche down hard: own one wedge competitors ignore — local/offline video gen for privacy-bound users ("Ollama for video") — and be the best at only that. A small defensible niche beats a broad indefensible platform.
  • Pick a vertical: stop being horizontal infra; become "text-to-video for [e-commerce product videos / D&D scenes / educational explainers]" where workflow + templates + domain data are the moat.
  • Be a library, not a platform: if the honest moat is "good abstraction code," ship the de-facto open SDK (LiteLLM-for-video) and let the ecosystem standardize on you — distribution as moat.
  • Accept no-moat, optimize for speed/reputation: treat it as a portfolio flagship and developer-reputation play, not a business. Legitimate and honest.
